## 一、已修复问题

### [高] Fix-1：CSS 校验器缺少对 `@layer`/`@container`/`@scope` 的限制

**文件：** `lib/css-validator.ts`

**原因：**
- `@layer` 可提升用户 CSS 的优先级，覆盖官方全局样式
- `@container` 容器查询作用于任意祖先元素，可超出 `.user-theme` 作用域
- `@scope`（CSS 草案）可显式绑定作用域，绕过选择器白名单

**修改：**
```ts
// 在 forbiddenAtRules 追加三项
"layer",
"container",
"scope",
```

### [高] Fix-2：`ThemeProvider.switchTheme` 在 apply 失败后用户主题样式丢失

**文件：** `components/ThemeProvider.tsx`

**原因：**
原逻辑是先 `removeStyle(旧版本)` 再 `await applyVersion(新版本)`。若 `applyVersion` 因网络错误或 hash 不匹配抛出异常，旧样式已被移除，页面裸奔至官方默认样式，且 `currentVersion` 状态不一致。

**修改：**
```ts
// 修改前
if (currentVersion) removeStyle(`user-theme-${currentVersion}`);
await applyVersion({ version: trimmed });

// 修改后：先记录旧版本，apply 成功后再移除旧 style
const prevVersion = currentVersion;
await applyVersion({ version: trimmed });
if (prevVersion && prevVersion !== trimmed) {
  removeStyle(`user-theme-${prevVersion}`);
}
```

### [高] Fix-3：`ThemeProvider.applyVersion` 使用 SSR 内联 CSS 时未校验 hash

**文件：** `components/ThemeProvider.tsx`

**原因：**
当 `preferSsr: true` 时，直接使用 `<style id="user-theme-xxx">` 内的 CSS 内容，并自行计算 hash 后存储。若页面受到 XSS 注入（攻击者修改了该 `<style>` 内容），则会直接使用篡改后的 CSS，且存入 IndexedDB 缓存，持久化攻击效果。

**修改：**
```ts
if (!input.expectedHash || hash === input.expectedHash) {
  await done(ssrCss, hash);
  return;
}
// hash 不匹配，降级走网络请求
```

### [中] Fix-4：`locale-store.listLocalePacks` N+1 串行 I/O

**文件：** `lib/server/locale-store.ts`

**原因：**
原实现用 `for...of` 循环串行 `await readJsonFile`，用户有 N 个语言包就会产生 N 次串行文件读取，响应时间随包数线性增长。

**修改：**
```ts
// 修改前：串行
for (const entry of manifest.packs) {
  const pack = await readJsonFile(...);
  results.push(...);
}

// 修改后：并行
const packFiles = await Promise.all(
  manifest.packs.map((entry) => readJsonFile(...))
);
return manifest.packs.map((entry, i) => ({ ...packFiles[i]... }));
```

### [中] Fix-5：`POST /api/user-i18n/packs` 未校验 translations 的值类型

**文件：** `app/api/user-i18n/packs/route.ts`、`app/api/user-i18n/packs/[packId]/route.ts`

**原因：**
仅检查 `translations` 是否为非数组对象，未验证每个 `value` 是否为 `string`。攻击者可传入 `{ key: { __proto__: ... } }` 或数值/对象，导致存储类型污染，后续 `Object.entries(pack.translations)` 等操作行为不确定。

**修改：**
```ts
// 逐值过滤，只保留 string 类型
const validated: Record<string, string> = {};
for (const [k, v] of Object.entries(raw)) {
  if (typeof v === "string") validated[k] = v;
}
```

### [中] Fix-6：`saveUserTheme` 版本数无上限，可持续写满磁盘

**文件：** `lib/server/theme-store.ts`

**原因：**
每次提交主题都追加一条版本记录，无任何上限限制。攻击者（或误操作用户）可循环提交，持续在 `.data/user-themes/` 下写入 CSS 和 JSON 文件，直至磁盘耗尽。

**修改：**
```ts
const MAX_VERSIONS_PER_USER = 50;

// 超出上限时淘汰最旧版本（按 createdAt 排序后取前 50）
const evicted = deduplicated.slice(MAX_VERSIONS_PER_USER);
const nextVersions = deduplicated.slice(0, MAX_VERSIONS_PER_USER);
for (const old of evicted) {
  await fs.unlink(getCssPath(...));
  await fs.unlink(getRecordPath(...));
  serverCache.delete(...);
}
```

## 二、建议修复（未改代码，需评估后处理）

### [高] 安全：用户身份鉴权基于 Cookie 原始值，无签名验证

**文件：** `lib/server/user-session.ts`

**问题：**
```ts
// 任何人可伪造 Cookie 或 Authorization Header 冒充任意用户
const fromCookie = parseCookieValue(cookieHeader, "userId");
const match = auth.match(/^Bearer\s+(.+)$/i);
if (match?.[1]) return normalizeUserId(match[1]);
```
`normalizeUserId` 只做 `trim()`，不做签名校验。生产环境应替换为 JWT 或 Session 签名机制。

**优先级：** 高（当前为 Demo 实现，若上线必须修复）

---

### [中] 安全：CSS 选择器白名单可通过 `.user-theme *` 传入 `display: none`

**文件：** `lib/css-validator.ts`

**问题：**
`isAllowedSelector` 允许所有以 `.user-theme` 开头的选择器，包括 `.user-theme *`。而 `isForbiddenDecl` 对 `display: none` 的检查只拦截了 `:root` 上下文，导致：
```css
.user-theme * { display: none; }  /* 通过校验，隐藏所有内容 */
```

**建议修复：**
```ts
if (prop === "display" && valueLower === "none") {
  return "禁止属性：display: none（可隐藏页面内容）";
}
```

---

### [中] 性能：`fetchJson` 在三个文件中重复定义

**文件：** `components/ThemeProvider.tsx`、`components/I18nProvider.tsx`、`components/LocaleManager.tsx`

**问题：** 完全相同的函数体被复制了三次。建议提取到 `lib/fetch-json.ts`：
```ts
export const fetchJson = async <T,>(url: string, init?: RequestInit): Promise<T> => { ... };
```

### [中] 性能/安全：API 无限流控

**文件：** `app/api/user-theme/route.ts`

**问题：** 除版本数上限外，提交频率无限制。每次提交都触发 PostCSS 解析（CPU 密集型）+ 文件 I/O。建议添加基于 `userId` 的请求频率限制（如 10 次/分钟）。

### [低] Bug：`LocaleManager.onExportJson` URL 对象可能提前撤销
